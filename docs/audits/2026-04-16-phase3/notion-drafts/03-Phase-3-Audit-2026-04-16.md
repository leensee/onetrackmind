### Pattern 1 ‚Äî Silent-drop data corruption handling

Stage 1 sweep confirmed 2 truly silent drops (`contextLoader.ts:120` styleExclusions JSON parse, `specLookup.mapRosterRow` malformed common_names) and 8 warn-and-drop (serializeMetadata variants, coerceSetting variants, etc.). Concentrated at the DB-boundary where unknown-shape values are coerced to typed values. No centralized coercion helper exists despite `typeUtils.ts` being designed for this purpose. Fix direction: extend `typeUtils` with `extractOneOf<T>` and `extractArray`; replace all direct casts and if-typeof chains with typed accessors.

### Pattern 2 ‚Äî Hardcoded edition/locale assumptions

Six locations outside `system-prompt.v1.0.ts` (which is intentionally edition-bound):
- `approvalGate.ts:16` ‚Äî `'leensee/onetrackmind'` GitHub repo
- `outputRouter.ts:197` ‚Äî `'OneTrackMind'` FCM notification title
- `expenseParser.ts:20` ‚Äî `'USD'` default currency
- `contextLoader.ts:38` ‚Äî `'America/Chicago'` default timezone
- `modelAudit.ts` audit prompt ‚Äî OTM v1 domain vocabulary
- `poGenerator.ts:98` ‚Äî `$` currency sigil in `formatCurrency`

All fixable via `EditionConfig` injection. Three are in the Tier 0+1 remediation scope (AG-2, S1-E2, MA-4).

### Pattern 3 ‚Äî Text heuristic where state-check or structured approach exists

Three instances, all in audit-adjacent code:
- `preflight.ts` autonomous-action detection ‚Äî phrase matching on output text
- `preflight.ts` gate-marker detection ‚Äî phrase matching on output text
- `expenseParser.ts` ‚Äî regex-based receipt parsing downstream of Claude vision extraction

Criterion #4 violation. Fix direction for preflight: check orchestrator state (did the decision tree route through a gate?) instead of inferring from output text. Fix direction for expenseParser (deferred to Tier 2): expand `ImageExtractorClient` interface to return structured `Partial<ExpenseRecord>` directly; fall back to regex only when structured extraction fails.

### Pattern 7 ‚Äî Reimplemented format logic for the same data type

Four instances formatting the same data shapes independently:
- `promptAssembler.ts` formats `ActiveFlag` for context injection
- `modelAudit.ts` formats `ActiveFlag` for audit prompt
- `preflight.ts` formats `ActiveFlag` for rule evaluation
- `orchestratorTools.ts` renders draft content via `JSON.stringify` for approval gate

Fix direction: shared `src/utils/formatters.ts` with `formatActiveFlag`, `formatOpenItem`, `formatMachineRef`, `formatDraftForApproval`. Folded into OT-9 Tier 0 remediation.

### Pattern 10 ‚Äî Magic numbers with no calibration tie-back

Five+ constants with comments justifying them but no test asserting the value or Phase 9 calibration entry pointing at them:
- `CHARS_PER_TOKEN = 4` (`tokenUtils.ts`)
- `+ 4` message overhead (two different usages, same literal)
- System prompt token count `~2,727` (claim only, no assertion)
- Preflight phone-number window `12/15`
- Model audit response cap `500 tokens`

These are Phase 9 calibration inputs. Recommendation: explicit calibration-target list added to Phase 9 Roadmap deliverables so these constants have a scheduled validation pass.
